# Pixiv 漫画翻译项目经验教训

> 一个从零开始的日漫翻译工具链，历经多轮迭代后总结的实战经验。
> 仓库：<https://github.com/Napoleoncqf/pixiv-manga-translator>

## 项目背景

基于 Saber-Translator 后端，搭建一个全自动的 Pixiv 日漫下载 + 翻译嵌字工具。目标是把某个 Pixiv 作者的整个 series（42 张漫画）完整翻译成中文。

从部署原项目到最终可用，大约经历了十几轮方案迭代。以下是核心经验。

---

## 一、架构决策：什么方案有效，什么无效

### ❌ 原方案：OCR → 文本翻译 → LAMA 修复 → 审查重试

一开始沿用 Saber-Translator 的完整管线：
1. CTD 检测气泡
2. MangaOCR 识别日文文字
3. 把 OCR 文本送 Gemini 翻译
4. LAMA 图像修复擦除原文
5. Saber 渲染中文文字
6. Gemini 视觉审查
7. 不合格自动重试（3 级：`dilate 10 → 25 → 50`，`lama_mpe → litelama`）

**踩过的坑：**

- **LAMA 对 JPEG 图的复杂背景处理很差**。漫画里大量的"浮动文字"（无边框内心独白写在人物头发上）擦不干净，三轮重试也擦不掉。
- **MangaOCR 对数字/标点容易错**。`37.3度` 被识别成 `3度3分`，整个气泡翻译就变成"3度3分"。
- **审查-重试循环治标不治本**。审查模型发现残留后，用相同算法重试更激进的参数，结果大同小异，只是重试 3 次浪费时间。最严重一次 42 张图跑了 2 小时，通过率 28%，大量 `_REVIEW.png` 堆在目录里没人管。
- **"白底兜底"补丁越补越丑**。最后一级失败时画白色圆角矩形覆盖，用 bbox 没法贴合椭圆气泡，导致遮挡气泡边线甚至人物。

### ✅ B+B 方案：Vision 翻译 + 背景色填充（最终方案）

重构后的新管线：
1. CTD 检测气泡（兼容保留）
2. **Gemini Vision 补检**：高分辨率扫描 CTD 漏的区域
3. MangaOCR（仅用于过滤 SFX / 非日文）
4. **Vision 直接翻译**：把每个气泡的 crop 和整页带编号标注图一起发给 Gemini，让它看图翻译
5. **采样气泡背景色**填充 polygon（不再用 LAMA）
6. 渲染中文译文

**为什么变简单反而变好：**

- **Vision 看图翻译比 OCR+翻译准确得多**，尤其是数字和场景依赖的内容。`37.3度` 不会再错成 `3度3分`。
- **双输入（整页+crop）给足上下文**。整页带红色编号标注让模型看到完整场景，crop 提供文字细节。这个组合对"卡牌游戏中的 3"、"体温计上的 38.3" 这类场景理解至关重要。
- **polygon 白底 + 背景色采样比 LAMA 可靠 100%**。LAMA 在复杂背景上会留残影，而直接用采样的背景色填充 polygon 永远干净。气泡内本来就是纯色，所以视觉上完全无违和。
- **不再需要重试循环**。42 张一次跑通，没有一张 `_REVIEW.png`。

**核心教训**：**生成式 AI 的能力边界变了，老管线的每一步都值得重新审视。** LAMA 是 2021 年的方案，在今天 Vision 模型直接看图翻译面前已经被降维打击。

## 二、调 AI 的具体踩坑

### 1. Pro 3.1 是 reasoning 模型，`max_tokens` 要留足

最严重的一次 bug：一张 14 气泡的复杂图，Pro 返回空 content，日志显示 `finish_reason=length`。调试发现：

```python
# 错误配置
max_tokens=3000  # 不够，reasoning 吃光后输出为空

# 正确配置
max_tokens=12000  # Pro 3.1 reasoning 消耗大量 token
```

**教训**：Reasoning 模型的 token 预算要按"思考 + 输出"一起算，不能按普通模型的经验值给。

### 2. Gemini 的 JSON 响应常被 markdown fence 包裹

Pro 和 Flash 都会返回：
````
```json
{...}
```
````

要先剥掉 fence 再解析。另外响应可能被截断（finish_reason=length），需要补齐未闭合的 `{` `[`。

### 3. 每日配额是个真实约束

Gemini 3.1 Pro 免费层每日 250 次请求。一次跑 42 张，每张 2-3 次调用（翻译+检测+审查），很快就会 429。应急方案：
- 切换到 Flash 保底
- Pro/Flash 质量差距不小，Flash 翻译容易犯错（如 `何もしてないでしょ` → "什么都行"）
- 第二天配额恢复后用 Pro 重跑

**教训**：做批量任务前先估算调用数 × 模型单价/配额，不要跑到一半才发现超配额。

### 4. Vision 模型对 OCR 的能力有边界

Gemini Vision 对普通字体的日文读取非常准确。但：

- **强风格化艺术字**（毛笔体、花体、装饰性的 `スマホ禁止！`）识别很差，经常"猜"出一个上下文相关但字面错误的翻译
- **极小字体**（气泡 < 30px）在缩略图里看不清，会漏检
- **密集场景里的小气泡**需要高分辨率（2400px+）才能看清

**教训**：
- 对艺术字这种 Vision 也看不清的东西，**不如不翻**（保留原图），硬翻反而出错
- 分辨率是关键变量，不要随便压缩

### 5. Vision 审查的非确定性

同一张图跑两次审查，判定可能不同。第一轮通过的图第二轮可能被标记为失败，反之亦然。所以"审查通过率"不是稳定指标，不能用来严格判定质量。

**教训**：审查只用来**识别严重问题**（漏翻、残留日文），不要迷信具体的通过数字。

## 三、Pipeline 设计原则

### 1. "宁可漏报，不要错译"

对于模型看不清/看不懂的内容：
- **保留原图**比**强行输出一个错译** 好得多
- 用户能自己读日文汉字，但乱译会完全误导

具体实现：
- `is_bubble_text()` 过滤掉 SFX、标签、艺术字 SFX
- Vision 返回空字符串时跳过擦除和渲染
- 空译文不触发白底，原图保留

### 2. 尊重漫画约定

漫画有固定的视觉习惯，代码要跟上：
- **中文默认竖排**（漫画阅读习惯），只有含数字/短文本/横扁气泡才横排
- **数字强制横排**（避免小数点在竖排中变成独立一行）
- **角色标签保留日文汉字**（`長女`/`次女`/`三女` 中文读者能直接看懂）
- **拟声词保留原样**（`ドキ`/`ガチ`，翻译反而破坏画面节奏）

这些规则看起来琐碎，但每条都是从某张失败图总结出来的。

### 3. 检测比翻译更重要

项目后期发现：**大部分"翻译失败"其实是检测失败**。CTD 漏掉了某个气泡，那个气泡的文字就不会被翻译，更不会被擦除。用户看到原日文残留，以为是翻译问题，实际是检测问题。

解决办法：
- Vision 补检（2400px + 艺术字专项扫描）
- 双通道合并（CTD + Vision 去重）
- 检测失败的代价远高于翻译失败的代价

### 4. 断点续跑是批量任务的必备特性

42 张图跑到一半脚本崩溃（各种原因：单条 API 超时、某张图异常），前面的工作都白费。所以：
- 检查输出目录已存在的文件自动跳过
- 单张图的异常用 try/except 隔离，不影响整批
- 每张图完成后立即保存，不等全部结束

## 四、Git 和部署的教训

### 1. 别往上游仓库推

最初 remote 指向 Saber-Translator 上游，直接 push 当然失败。建了自己的 fork 才推成功。记得：
- 保留 `upstream` 指向原仓库以便拉更新
- `origin` 指向自己的 fork

### 2. 仓库大小会成为 push 障碍

上游项目历史里塞了 200MB+ 的字体和前端构建产物。第一次 push 反复在 1GB 左右断连。最后用 `--orphan` 新建了一个干净分支丢弃历史，`.git` 从 186MB 降到 6.7MB，push 秒成。

**教训**：如果不需要历史，`--orphan` 一个干净的 initial commit 比挣扎着 push 大仓库更快。

### 3. `.env` 必须在 `.gitignore` 第一天就加

我运气好，Claude Code 在启动前主动加了 `.env` 到 `.gitignore`。否则 API key 就会随第一次 commit 公开到 GitHub 上。提交前用 `git log --all -p | grep -E "AIzaSy|sk-"` 扫一遍是好习惯。

## 五、与 AI 协作的工作流

### 1. 问题必须具体到图，不能模糊

```
❌ "有些图翻译得不好"
✅ "137178529_p0 第三格右上气泡漏翻"
```

前者让我到处猜哪张图有问题，后者可以直接定位处理。用户每次发具体文件名 + 截图，问题修复速度快一个数量级。

### 2. 迭代节奏：一次只改一层

每轮修复只改一个维度：
- 第一轮：修复编码（译文乱码）
- 第二轮：修复方向（默认竖排）
- 第三轮：修复擦除（polygon 背景色）
- 第四轮：修复检测（高分辨率 Vision）

不要一次改 5 个地方，否则出问题没法定位。

### 3. 用小样本测试再推广

每次改完先在 5 张代表性图上跑，看效果没问题再推广到全部 42 张。避免一次改砸 42 张要等几小时重跑。

### 4. 定时检查 + 后台任务

批量任务动辄 30-60 分钟，不要干等。设定 `ScheduleWakeup` 定期回来看进度，期间做其他事。

## 六、数据：从乱到可用的进化

| 阶段 | 方案 | 通过率 | 耗时 |
|------|------|--------|------|
| 初版 | LAMA + 重试 + 审查 | ~28% | 2h |
| B+B 重构 | Vision 翻译 + polygon 白底 | ~95% | 50min |
| 增强检测 | +Vision 补检 2400px | ~95% | 55min |
| 默认竖排 + 标签过滤 | 漫画约定 | ~95% | 55min |
| Pro 重跑 + 艺术字过滤 | 最终版 | **~98%** | 50min |

**"通过率"** 这里指视觉审查没发现严重问题的比例。剩余 2% 多是**强风格化艺术字**，属于模型能力上限，保留原图即可。

## 七、核心心得（一句话版本）

1. **生成式模型的能力边界每半年就变**，老架构的每一步都值得重新审视。
2. **检测失败 > 翻译失败**，把资源投在检测准确度上回报最高。
3. **Reasoning 模型的 max_tokens 要给够**（至少 8000+）。
4. **宁可不翻译也不要错译**，保留原图是合格的兜底。
5. **漫画默认竖排**，不是 bug 是约定。
6. **分辨率是 Vision 准确度的 bottleneck**，能给高就给高。
7. **`.env` 进 gitignore，API key 扫一遍再 push**。
8. **断点续跑是批量任务的必备设计**，不要指望一次跑通。

---

## 八、还没解决的问题

诚实地列一下 Pipeline 达不到的地方：

1. **强风格化艺术字**（`スマホ禁止！` 毛笔体、带运动线的装饰字）：Vision 也看不准，只能保留原图
2. **极密集小气泡渲染拥挤**：Saber-Translator 渲染器的限制，auto font size 给的字号偏小
3. **OCR 混入相邻标签**：有时 Gemini 会把相邻的 `三女` 前缀附到翻译里（已用过滤词解决大部分）
4. **翻译风格的主观性**：Pro 和 Flash 对同一句话可能给出不同风格的翻译，审查模型也会给不同判定

这些都不是代码 bug，是生成式 AI 当前能力的真实边界。

## 附：Pipeline 核心常量

```python
# 关键参数
TRANSLATE_MODEL = "gemini-3.1-pro-preview"  # Pro 质量好但有配额，Flash 兜底
FONT = "ZhanKuKuaiLeTi.ttf"                 # 站酷快乐体，圆润可爱适合漫画
MAX_TOKENS_VISION = 12000                    # Pro reasoning 占大量 token，要给够
VISION_DETECT_MAX_DIM = 3500                 # 检测分辨率，越高越能看清小气泡

# 过滤规则
SFX_WIDTH_THRESHOLD = 45      # 宽 < 45 且 ≤ 2 字符视为音效字
ART_AREA_PER_CHAR = 5000       # 每字面积 > 5000 px 视为艺术字 SFX
ROLE_LABEL_WORDS = {"長女", "次女", "三女", ...}  # 角色标签保留原图
```

这些数字都是从失败案例里调出来的，不是拍脑袋定的。
